# Side-by-side test: plain one-shot vs. the review step (3 tasks)

**Date**: 2026-06-13 · **Same model both ways. Nothing was set up to fail.** Three tasks that
teams build often. For each one: first write the code in a single quick pass, with the tests
a quick pass would naturally write (the "plain" version). Then run the framework's
rule-based tests against the **same, unchanged** code (the "review" version).

## Result — the review step found a real bug in all three; how serious varied

| Task | Plain version | What the review step found | How serious |
|------|---------------|----------------------------|-------------|
| **Compare versions** | 3/3 tests passed | 4 bugs, including **`1.0.0-rc.1` treated as newer than `1.0.0`** | **High** — would ship a pre-release over the real release |
| **Split money** | passed on even amounts | `split($10, 3)` → `[333, 333, 333]` cents = **$9.99**, not $10 | **High** — the money does not add up |
| **Pagination** | 3/3 tests passed | `page = -1` quietly returns **the wrong rows** with no error; `pageSize 0` returns nothing | **Medium** — may not matter for an internal helper |

Every plain version looked done: its own tests passed and it was ready to merge. The review
step's tests — written from the rules — failed on the same code and showed the bugs.

## What this shows — honestly

- **It is not a one-off.** Three different kinds of task, three real catches. The version and
  money bugs would cause a real production problem. The pagination ones are smaller, and some
  teams might not block on them.
- **It helps most where the task has tricky cases.** All three tasks have hidden edge cases —
  that is why they were picked, and they are normal everyday work. On a truly simple task done
  right, the review step finds **nothing**. The framework is not magic; it pays off where edge
  cases hide.
- **Why the plain version missed them:** same model both times. The plain code was not bad.
  Nothing made it think about the edge cases. "Write me X" does not remind you about
  pre-release version order or splitting money to the exact cent. The spec step and the
  one-test-per-rule check turn "I forgot" into a failed review instead of a silent bug.

## In short

On three everyday tasks, a good quick pass shipped real bugs with passing tests, and the
review step caught them before merge — twice at "this would break production" level. That is
the case for the review step. The honest limit: on a task with no hidden cases it finds
nothing, so use it where it helps (and use `--micro` or plain Copilot when there is little to guard).

## Reproduce
Each task is two short files (TypeScript + `node:test`). Run `npm test`: the plain tests
pass; the rule-based tests fail on the unchanged code until you fix it.
